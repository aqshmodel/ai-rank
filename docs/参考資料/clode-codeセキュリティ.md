



【ClaudeCode初心者向け】誰でもできる最低限のセキュリティ対策手法：18選 
ClaudeCode初心者向け誰でもできる最低限のセキュリティ対策手法18選
Claude Code（AnthropicのターミナルAIエージェント）を使い始めた方が、機密情報の漏洩・ファイルの誤削除・意図しないコマンド実行といった事故を防ぐための対策を18個まとめました。
「エンジニアじゃないけどClaude Codeを使いたい」という方を想定しています。難しい知識は不要で、設定ファイルのコピー・コマンド1つ・Finder操作のレベルで全部できます。
対策1｜専用フォルダだけで起動する
ホームディレクトリ全体をClaude Codeに触らせないようにします。
ホームディレクトリ（~）で claude を起動すると、デスクトップ・ドキュメント・写真など全ファイルにアクセスできる状態になります。作業用の空フォルダを1つ作ってその中だけで起動するだけで、アクセス範囲を大幅に限定できます。
やり方：

mkdir ~/my-claude-project
cd ~/my-claude-project
claude
Finderで新しいフォルダを作り、そこでターミナルを開いてから claude と打つだけです。
対策2｜.envファイルをプロジェクト内に絶対置かない
.env ファイルにはAPIキーやパスワードが入っています。これをプロジェクトフォルダ内に置くと、Claude Codeがアクセスできる状態になります。
.gitignore や .claudeignore に書いても完全には防げないケースが報告されています。根本解決は「プロジェクトフォルダの外に置く」か「そもそも作らない」です。
やり方：
* 		.env ファイルはプロジェクトフォルダの外の別フォルダに移動する
* 		可能であれば1Password・macOSキーチェーン・環境変数の手動設定を使う
対策3｜Claudeの学習データ使用をオフにする
入力した内容がAnthropicのモデル学習に使われないよう設定します。
やり方：
* 		claude.ai/settings/privacy を開く
* 		「Claudeの改善にご協力ください」をオフにする
Pro・Teamプランでも初期状態ではオンになっているため、必ず確認してください。
対策4｜Plan Modeで作業計画を先に確認してから実行する
Claude Codeには実行前に計画だけを表示する「Plan Mode」があります。ファイルの変更やコマンド実行の前に「何をするつもりか」をClaude Codeに説明させることで、意図しない操作を事前に防げます。
やり方：
Claude Codeに指示を出す前に、まず「どういう手順でやるか説明して、実行はしないで」と伝える習慣をつけましょう。
チェック観点：
* 		触るファイルの一覧に見覚えのないものがないか
* 		実行するコマンドに rm・curl・cat .env が含まれていないか
* 		想定外のディレクトリにアクセスしようとしていないか
「計画を確認してGOサインを出す」この一手間で9割の事故を防げます（公式ベストプラクティス）。
対策5｜--dangerously-skip-permissionsフラグを絶対に使わない
Claude Codeは通常、ファイル操作やコマンド実行のたびにユーザーの承認を求めます。--dangerously-skip-permissions フラグはその承認をすべてスキップする設定です。
名前に「dangerously（危険）」と書いてある通り、このフラグを使うとClaude Codeがすべての操作を無承認で実行できるようになります。慣れてきて「毎回確認が面倒」と感じても、このフラグだけは使わないようにしましょう。
対策6｜/permissionsで現在の権限ルールを定期的に確認する
Claude Code内で /permissions と入力すると、現在どんな許可・拒否ルールが設定されているかを一覧で確認できます。

/permission
設定したつもりのdenyルールが反映されていない、見覚えのないallowルールが追加されているといった状態を定期的にチェックできます。プロジェクトを始める前に1回確認する習慣をつけると安心です。
対策7｜CLAUDE.mdにセキュリティルールを明記する
プロジェクト直下に CLAUDE.md ファイルを作成してセキュリティルールを書いておくと、Claude Codeが起動時に自動で読み込み、高優先度で守ります。
やり方：
プロジェクトフォルダに CLAUDE.md という名前のテキストファイルを作り、以下をコピーして保存します。

## セキュリティ最優先ルール（絶対遵守）
- .envや機密ファイル（secrets, credentials, token等）を一切読まない・出力しない
- 危険コマンド（rm -rf, curl 外部URL, cat .env など）は実行前に必ず確認を求める
- 機密情報はユーザー自身に確認させる
- ホームディレクトリ（~/）への直接アクセスは禁止
設定ファイルによるブロックと合わせて使うことで、多層防御になります。
対策8｜Gitで作業前にコミットしてロールバックポイントを作る
Claude Codeに大きな変更を任せる前に、必ず git commit でその時点の状態を保存しておきます。万が一ファイルが壊れたり消えたりしても、コミット時点に戻せます。
「Claude Codeに作業を依頼する直前にコミット」を習慣にするだけで、ほぼすべてのファイル破壊事故から回復できます。
対策9｜信頼できないリポジトリをそのままcloneしない
外部からダウンロードしたコードには、セキュリティ設定を書き換えるファイルや悪意のあるスクリプトが含まれている場合があります（Check Pointが2025年に報告した脆弱性パターン）。
やり方：
* 		外部のリポジトリをそのままcloneしてClaude Codeを起動しない
* 		不明なコードを試したい場合は、空のフォルダからClaude Codeに「新規で作って」と指示する
* 		どうしてもcloneが必要な場合は、中の .claude/ フォルダと CLAUDE.md を確認してから使う
対策10｜AnthropicのAPIキーに使用量の上限を設定する
Claude Codeを自分のAPIキーで動かしている場合、キーが漏洩すると第三者に使われてAPIの課金が発生します。Anthropicの管理画面で使用量の上限（Spending Limit）を設定しておくと、万が一の被害を最小化できます。
やり方：
* 		console.anthropic.com にログイン
* 		「Settings」→「Billing」→「Spending Limits」
* 		月間の上限金額を設定する
Claude Code専用のAPIキーを発行して他のサービスと分けておくと、万が一のときに該当キーだけを無効化できます。
対策11｜パスワードや認証情報をClaude Codeのチャットに直接入力しない
「このAPIキーで接続して」「このパスワードでログインして」のように認証情報をチャットに直接打ち込むと、会話履歴として記録されます。
やり方：
* 		パスワードやAPIキーをチャットに貼らない
* 		環境変数でターミナルに設定してから「環境変数から読み取って」とClaude Codeに指示する
Claude Codeは環境変数を読み取ることができるので、チャットに直接書かなくても認証情報を渡せます。
対策12｜MCPサーバーは使うものだけを有効にする
MCP（Model Context Protocol）はClaude CodeをSlack・Google Drive・データベースなど外部サービスと連携させる仕組みです。便利な反面、接続しているサービスが多いほど情報が渡る範囲が広がります。
やり方：
* 		~/.claude/settings.json の mcpServers に書いているサービスのうち、今使っていないものを削除する
* 		新しいMCPサーバーを追加するときは、何ができるサーバーかを確認してから追加する
定期的に「本当に今使っているか？」を見直す習慣をつけましょう。
対策13｜/clearで定期的にコンテキストをリセットする
Claude Codeは会話の履歴を「コンテキスト」として保持します。長時間の作業を続けると、以前の会話内容（ファイルの内容・設定情報など）がコンテキストに残り続けます。
新しい作業を始めるときや、機密性の高い内容を扱った後は /clear でリセットするのが安心です。
コンテキストをクリアするだけで、前の会話内容がClaude Codeに見えなくなります。
対策14｜Claudeが作成・変更したコードは必ず目視してからcommitする
Claude Codeが生成したコードは自動的に動くように見えますが、意図しない処理が含まれている可能性があります。動作確認だけでなく、コードの中身も確認してからGitにコミットする習慣をつけましょう。
確認のポイント：
* 		見知らぬURLへのアクセスが含まれていないか
* 		外部サービスへのデータ送信処理が含まれていないか
* 		rm のような削除処理が想定外の場所に使われていないか
GitHub Desktopの「Changes」タブで差分を確認するだけでも、大きなミスを防げます。
対策15｜外部URLをそのままClaude Codeのプロンプトに貼り付けない
「このURLのページを読んで処理して」とURLを貼ると、そのページにプロンプトインジェクション（ページの内容にAIへの不正な命令が埋め込まれている）が含まれている場合があります。
やり方：
* 		不明なサイトのURLをそのままClaude Codeに渡さない
* 		URLを使う場合は事前にブラウザで内容を確認してから渡す
* 		「このURLの内容を要約して」より「この内容をテキストで貼るから要約して」のほうが安全
対策16｜作業ログを残して何をさせたか記録する
Claude Codeに何をさせたかの記録がないと、問題が起きたときに原因を追えません。簡単なメモで構いません。
やり方：
* 		プロジェクトに WORKLOG.md を作り、日付と作業内容を1行ずつ書く
* 		Claude Codeに「今日の作業をWORKLOG.mdに追記して」と頼むと自動化できる
* 		Gitのコミットメッセージを詳しく書くことでも代用できる
「いつ・何をさせたか」が残っていると、問題の切り分けが格段に楽になります。
対策17｜claude.ai/settingsでアクティブセッションを定期的に確認する
claude.aiの設定画面では、現在ログイン中のセッション（デバイス・ブラウザ）を一覧で確認できます。見覚えのないセッションがあれば、即座に無効化できます。
やり方：
* 		claude.ai/settings/security を開く
* 		アクティブセッションの一覧を確認する
* 		見覚えのないセッションは「ログアウト」する
月1回確認するだけで、アカウント乗っ取りの早期発見につながります。
対策18｜Claude Codeを最新バージョンに保つ
Claude Codeはセキュリティ修正を含むアップデートが定期的にリリースされます。古いバージョンを使い続けると、修正済みの脆弱性にさらされたままになります。
月1回か、Claude Codeを起動したときにバージョン確認のメッセージが出たタイミングで実行する習慣をつけましょう。

AIエージェント開発では、**「モデルの賢さ」より「権限の絞り方」と「秘密情報の守り方」**が重要です。Claude Code、Cursor、Antigravity で共通して意識すべき対策は、最小権限・秘密情報の分離・サンドボックス・変更レビュー・外部通信の監視です。 [ses-base](https://ses-base.com/articles/google-antigravity-vulnerability-security-guide/)

## 共通で必須の対策

- **権限は最小限にする。** Claude Code は既定で読み取り中心で、編集やコマンド実行には明示承認を求める設計です。 [code.claude](https://code.claude.com/docs/en/security)
- **秘密情報を作業ディレクトリに置かない。** `.env`、鍵ファイル、Firebase設定、keystore などは読み取り禁止または厳重管理にし、会話やログに貼らない運用が推奨されています。 [zenn](https://zenn.dev/tukiyubi/articles/7f43f23f053394)
- **外部通信をむやみに許可しない。** `curl` や `wget` のような任意取得系コマンドは危険で、Claude Code でも既定でブロック対象になっています。 [code.claude](https://code.claude.com/docs/en/security)
- **AIの変更をそのまま信用しない。** `git diff`、テスト、静的解析で、想定外の改変や脆弱な実装が入っていないか確認します。 [satsuki-bistro](https://satsuki-bistro.com/ai-agent-security-claude-code-guide/)
- **サンドボックスや分離環境を使う。** 失敗時の被害を限定するため、開発用PCと本番資産を分けるのが有効です。 [note](https://note.com/genies/n/nfd7672b05d94)

## Claude Code で気をつけること

Claude Code は権限ベースの設計で、機密操作には承認が必要です。 [code.claude](https://code.claude.com/docs/en/permissions)
実務では、`.env*` や署名鍵、`google-services.json` などを読み取り禁止にして、環境変数の出力や危険な削除コマンドも制限する運用が有効です。 [zenn](https://zenn.dev/tukiyubi/articles/7f43f23f053394)
また、`settings.json` と `CLAUDE.md` の両方でルールを明文化し、誤って機密を貼る事故も防ぐのがよいです。 [note](https://note.com/genies/n/nfd7672b05d94)

## Cursor で気をつけること

Cursor は **Privacy Mode** を有効にすると、コードやチャット内容が保存・学習されない設計です。 [blog.cloudnative.co](https://blog.cloudnative.co.jp/26826/)
そのため、機密案件では Privacy Mode を前提にし、追加でOS側の権限も最小限に絞るのが安全です。 [zenn](https://zenn.dev/kimkiyong/articles/e1d20233f468b9)
また、第三者提供の設定や拡張機能は中身を理解できない限り入れないことが推奨されます。 [note](https://note.com/famous_eel8721/n/n94be4c6a7ca4)

## Antigravity で気をつけること

Antigravity では、信頼できないリポジトリや入力に対する **プロンプトインジェクション** が大きなリスクです。 [pillar](https://www.pillar.security/blog/prompt-injection-leads-to-rce-and-sandbox-escape-in-antigravity)
報告例では、サンドボックスや Secure Mode があっても、ツール呼び出し経路の弱点から迂回される可能性が示されています。 [pillar](https://www.pillar.security/blog/prompt-injection-leads-to-rce-and-sandbox-escape-in-antigravity)
そのため、ワークスペース内の入力を鵜呑みにせず、ネットワーク制限、書き込み範囲制限、実行ログ確認を組み合わせる必要があります。 [zenn](https://zenn.dev/tokyo_bulkup/articles/82a44be19b0105)

## 実務チェックリスト

- 機密ファイルはAIに読ませない。 [zenn](https://zenn.dev/tokyo_bulkup/articles/82a44be19b0105)
- 承認なしでのコマンド実行を許さない。 [code.claude](https://code.claude.com/docs/en/permissions)
- 外部通信系コマンドを制限する。 [code.claude](https://code.claude.com/docs/en/security)
- Privacy Mode / Secure Mode を有効化する。 [cursor](https://cursor.com/security)
- `git diff` とテストで変更を毎回確認する。 [ses-base](https://ses-base.com/articles/google-antigravity-vulnerability-security-guide/)
- 作業環境を本番環境から分離する。 [ses-base](https://ses-base.com/articles/google-antigravity-vulnerability-security-guide/)
- ログや会話に秘密を残さない。 [note](https://note.com/famous_eel8721/n/n94be4c6a7ca4)

必要なら次に、**「Claude Code / Cursor / Antigravity それぞれの設定項目を、すぐ使えるチェックリスト形式」**に整理できます。


AIエージェント機能を搭載した開発ツール「Claude Code」「Cursor」「Antigravity」「Codex」などを利用する際には、従来の開発ツールとは異なるセキュリティリスクを理解し、対策を講じることが不可欠です。これらのツールは自律的にコードを生成・修正し、コマンドを実行できるため、利便性が高い一方で、乗っ取られた際の被害が甚大になる可能性があります[2, 6, 19]。
最低限知っておくべきセキュリティ対策は、大きく「環境の隔離」「権限の管理」「データの保護」「サプライチェーンの管理」の4つに分類できます。
AIエージェントに共通する主要なセキュリティリスク
自律的に動作するAIエージェントは、従来のアプリケーションにはなかった「実行リスク」をもたらします。不正な指示によって、単に誤った情報が出力されるだけでなく、実際にファイルが操作されたり、APIが呼び出されたりするため、被害が実世界に及びます。
業界専門家が策定したセキュリティリスクランキング「OWASP Top 10 for Agentic Applications」では、以下のようなリスクが上位に挙げられています。
 * プロンプトインジェクション: 攻撃者が悪意のある指示を紛れ込ませ、AIエージェントの安全機能を迂回させ、意図しない動作をさせる[6, 8]。
 * ツールの悪用: 正規のツールやAPIを不正に利用し、情報漏洩やシステムの不正操作を行う。
 * 権限・IDの悪用: AIエージェントに与えられた権限を乗っ取り、アクセス権のないデータやシステムに侵入する[6, 8]。
 * 意図しないコード実行: 攻撃者が用意した悪意のあるコードをエージェントに実行させる。
これらのリスクを踏まえ、具体的な対策を実践する必要があります。
最低限実施すべきセキュリティ対策
1. 環境の隔離（サンドボックス化）
AIエージェントが万が一暴走しても、ホストOSや重要なシステムに影響が及ばないように、隔離された環境で実行することが最も重要な対策の一つです。
 * WSL2の利用: Windowsユーザーの場合、WSL2 (Windows Subsystem for Linux 2) を使用して、AIエージェントの実行環境をホストOSから隔離します。これはプロの現場における標準的な構成とされています。
 * DevContainerやDockerの活用: ツールごとに開発コンテナを用意することで、プロジェクトごとに環境を分離し、リスクを局所化できます。複数のAIエージェントが連携するシステムでは、サンドボックス化されたDocker環境での動作が推奨されています。
2. 権限の管理（最小権限の原則）
AIエージェントには、タスク遂行に必要な最小限の権限のみを与える「最小権限の原則」を徹底します[6, 8, 49]。
 * 実行権限の制限: root権限や管理者権限でAIエージェントを実行することは絶対に避けるべきです。
 * コマンド実行の確認: Claude Codeなどのツールには、コマンドを実行する際に毎回ユーザーに許可を求める設定（askモード）があります。git push のような影響の大きい操作や、安全性が確認できない操作については、この設定を有効にすることが推奨されます[2, 3]。
 * アクセス制御のカスタマイズ: deny設定を利用して、機密ファイルや重要な設定ファイルへのアクセスを明示的に禁止します。これにより、意図しない情報漏洩を防ぎます。
 * MFA（多要素認証）の利用: ツールへのログインには、YubiKeyなどのハードウェアセキュリティキーを含むMFAを設定したアカウントを使用し、アカウント乗っ取りのリスクを低減させます。
3. データの保護と情報漏洩対策
AIエージェントは、対話内容や読み込んだファイルを開発元のサーバーに送信して処理を行います。機密情報が意図せず外部に送信されるリスクを常に意識する必要があります。
 * 学習させない設定の徹底: Claudeなどのサービスでは、ユーザーが入力したデータをAIモデルの学習に利用させない設定が可能です。機密情報を扱う場合は、必ずこの設定をオフにしてください。
 * 機密情報の除外: AIにコードベースを読み込ませる前に、APIキーやパスワードが記載された.envファイルなどの秘匿情報が含まれていないか確認・除外します。
 * API経由での利用: より高度なセキュリティが求められる場合、Web UI版ではなくAPIを利用して自社環境内に閉じたチャットシステムを構築する方法があります。Amazon BedrockやGoogle Cloud Vertex AI経由でClaudeを利用すれば、データが自社の管理下から出ることなくAIを利用できます。
4. ツール・サプライチェーンの管理
AIエージェントが利用する外部ツールやライブラリ、モデル自体に脆弱性が存在する可能性があります。
 * ツールの信頼性検証: 利用するツールやプラグインの提供元を検証し、信頼できるもののみを使用します。
 * 脆弱性情報の監視: 利用しているツールの脆弱性情報は常時チェックが必要です。例えば、過去にはGoogle Antigravityが公開直後に脆弱性を指摘されたり[34, 35]、CursorやCodexにも脆弱性が見つかった事例が報告されています[13, 18, 26]。
 * シャドーAIの禁止: 会社が許可していないAIツールを従業員が勝手に利用する「シャドーIT（シャドーAI）」は重大なリスクです。組織として安全な利用環境を整備し、許可されたツールの利用を徹底させる教育が重要です。
これらの対策は、AIエージェントの強力な機能を安全に活用するための第一歩です。AIエージェントの普及に伴い、セキュリティの考え方も「ルールを運用する」仕事から「AIが守るべきルールを定義し、監査する」仕事へとシフトしていくでしょう。
